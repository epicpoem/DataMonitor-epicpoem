## 1. 프로젝트 배경

가상의 반도체 회사 "S-Semi"가 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless),
대학 연구실 등에 납품하는 시스템이다.
주문량 급증으로 엑셀/메모장 기반 관리의 한계가 드러나 체계적인 콘솔 기반 시스템을 개발한다.

--------------------------------------------------------------------------------
## 2. 시스템 개요

- 플랫폼    : 콘솔 기반 (CLI), 담당자가 직접 명령 입력
- 생산 라인 : 단일 라인, 시료를 하나씩 순차 생산 (FIFO)
- 생산 대상 : 주문이 들어온 시료에 대해서만 생산

### 역할 구성
| 역할 | 담당 업무 |
|------|-----------|
| 고객 | 필요한 시료를 이메일로 요청 (시료 요청자) |
| 주문 담당자 | 고객 요청을 받아 주문서 작성 및 주문 관리 |
| 생산 담당자 | 시료 등록, 주문 수신 후 승인 또는 거절, 시료 생산·승인 |

--------------------------------------------------------------------------------
## 3. 주문 상태 정의

모든 주문은 아래 5가지 상태 중 하나를 가진다.
REJECTED는 모니터링에서 제외되는 비정상 흐름이다.

| 상태 | 의미 |
|------|------|
| RESERVED  | 주문 접수 (최초 상태) |
| REJECTED  | 주문 거절 (모니터링 제외) |
| PRODUCING | 주문 승인 완료 + 재고 부족 → 생산 중 |
| CONFIRMED | 주문 승인 완료 + 재고 충분 → 출고 대기 중 |
| RELEASE   | 출고 완료 (최종 상태) |

### 주문 상태 전이 흐름
```
[RESERVED]
     ↓ (승인/거절 판단)
    ├─ 거절 → [REJECTED]
    └─ 승인 → 재고 확인  ← (구두설명) 이 단계에서만 실시간 처리 적용
                │           물리적 재고(보유 재고 + PRODUCING 진행분 정률 합산)로 판단
                ├─ 재고 충분 → [CONFIRMED] → 출고처리 → [RELEASE]  ※ RELEASE 시 재고 차감
                └─ 재고 부족 → [PRODUCING] → 생산완료 → [CONFIRMED] → 출고처리 → [RELEASE]
```

## 4. 메인 메뉴 구성

| 번호 | 메뉴명 | 기능 요약 |
|------|--------|-----------|
| [1] | 시료 관리 | 시료 등록 / 목록 조회 / 이름 검색 |
| [2] | 시료 주문 | 고객 주문 접수 (RESERVED 생성) |
| [3] | 주문 승인/거절 | RESERVED 주문 목록 확인 후 승인·거절 처리 |
| [4] | 모니터링 | 상태별 주문 수 + 시료별 재고 현황 |
| [5] | 생산라인 조회 | 현재 생산 중인 시료 + 대기 큐 확인 |
| [6] | 출고 처리 | CONFIRMED 주문 출고 실행 → RELEASE 전환 |

메인 화면에는 시스템 현황 요약 표시:
  - 등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수

## 5. 기능 명세

### 5-1. 시료 관리 (Sample Management)

시료는 시스템의 기본 단위. 등록된 시료만 주문 가능.

#### 시료 속성 (데이터 모델)
| 필드 | 설명 |
|------|------|
| 시료 ID | 고유 식별자 (예: S-001) |
| 이름 | 시료 명칭 (예: 실리콘 웨이퍼-8인치) |
| 평균 생산시간 | 1개당 생산에 걸리는 평균 시간 (단위: min/ea) |
| 수율 | 정상 시료 / 총 생산 시료 (예: 0.92 = 92%) |
| 현재 재고 | 현재 보유 수량 (ea) |

수율 예시: 100개 생산 중 정상품 90개 → 수율 = 0.9

#### 하위 메뉴
| 메뉴 | 기능 |
|------|------|
| 시료 등록 | 새 시료 추가. 속성(ID, 이름, 평균생산시간, 수율) 입력. 초기 재고는 0으로 설정됨. ID 중복 시 재입력 유도 |
| 시료 조회 | 전체 시료 목록 + 현재 재고 수량 표시 |
| 시료 검색 | 속성(ID, 이름, 수율 등) 선택 후 검색 가능. 검색 대상 속성을 선택할 수 있도록 구현 |

---

### 5-2. 시료 주문 (Order Placement)

고객 요청을 받아 주문 담당자가 주문을 생성.

#### 입력값
- 시료 ID
- 고객명
- 주문 수량

#### 처리 결과
- 주문번호 자동 생성 (예: ORD-20260416-0043)
  - 형식: ORD-{날짜}-{날짜별 순번 4자리} (일별 주문 10,000건 미만 가정)
- 주문 상태: RESERVED로 설정
- 재고 확인은 승인 메뉴([3])에서 별도 수행

---

### 5-3. 주문 승인/거절 (Order Approval)

RESERVED 상태의 주문 목록을 조회하고 승인 또는 거절 처리.

#### 재고 판단 기준 (정책)
- 재고 확인은 실제(물리적) 재고를 기준으로 한다.
- PRODUCING 중인 수량은 물리적 재고에 정률(시간 비례)로 실시간 산입한다.
  물리적 재고(실시간) = 보유 재고 + 각 PRODUCING 주문의 진행분 비례 합산
- 부족분 = 주문 수량 - 물리적 재고(실시간 반영)

#### 승인 처리 로직 (자동 분기)
```
승인 요청
  → 재고 확인
      ├─ 재고 >= 주문수량 → 즉시 CONFIRMED 전환
      └─ 재고 < 주문수량 → 생산라인 자동 등록 + PRODUCING 전환
```

재고 부족 시 화면 표시 예:
  "재고 부족. 부족분 170 ea 승인하시겠습니까? (실생산량 206 ea / 165 min)"

#### 물리적 재고 실시간 반영 계산 (구두설명)
- 승인 시 재고 확인은 PRODUCING 중인 진행분을 물리적 재고에 실시간(시간 비례)으로 포함하여 계산한다.
- 공식:
  ```
  PRODUCING 주문 1건의 진행분 = 실생산량 * (경과 시간 / 총 생산 시간)
                                단, 경과 시간 >= 총 생산 시간이면 실생산량 전체

  물리적 재고(실시간) = 보유 재고 + 각 PRODUCING 주문의 진행분 합산
  부족분             = 주문 수량 - 물리적 재고(실시간)
  ```
- 물리적 재고(실시간) >= 주문 수량 → CONFIRMED 전환
- 물리적 재고(실시간) < 주문 수량  → 부족분만큼 생산라인 등록 후 PRODUCING 전환

#### 예정 입고 수량 계산 방식 (구두설명)
- 예정 입고 수량은 생산 시간에 비례하여 계산한다.
- 모든 생산은 일정한 속도로 이루어지며, 시간에 비례하여 제품이 생산되는 것으로 가정한다.
  ```
  예정 입고 수량 = 실생산량 * (경과 시간 / 총 생산 시간)
  ```
- 경과 시간이 총 생산 시간 이상인 경우 실생산량 전체를 예정 입고 수량으로 처리한다.

#### 생산 여부 및 생산 수량 결정 시점 (구두설명)
- 생산 여부(재고 충분/부족 분기)는 **주문 승인 시점에만** 판단한다.
- PRODUCING 또는 CONFIRMED로 전환된 이후에는 재고 변동이 발생해도 해당 주문 상태를 소급 재계산하지 않는다.
- 생산 수량 역시 **주문 승인 시점에 계산된 실 생산량으로 고정**된다.
- 생산 진행 중 실제 생산량이 달라지더라도 승인 시점에 결정된 수량대로 진행하며, 생산 중 추가로 생산된 수량은 주문 시점에 반영된 내용이 아니므로 무시한다.

#### 재고 차감/추가 시점 (정책)
- 승인 → CONFIRMED 전환 시: 재고 차감 없음
- 출고 처리(RELEASE) 완료 시: 주문 수량만큼 물리적 재고에서 차감
- PRODUCING 진행 중: 생산 진행분이 물리적 재고에 실시간(정률) 반영됨

#### 거절 처리
- 즉시 REJECTED 상태로 전환
- 모니터링에서 제외됨

---

### 5-4. 모니터링 (Monitoring)

#### 주문량 확인
- 상태별 주문 건수 표시: RESERVED / CONFIRMED / PRODUCING / RELEASE
- REJECTED는 표시하지 않음

#### 재고량 확인
각 시료별 재고 수량 + 재고 상태 표기

| 상태 | 조건 |
|------|------|
| 여유 | 주문 대비 재고 충분 |
| 부족 | 주문 대비 재고 수량 부족 |
| 고갈 | 재고 수량이 0 |

※ "주문 대비 재고" 판단 기준: **CONFIRMED + PRODUCING** 상태 주문 총량과 비교
   (RESERVED는 REJECTED 처리될 수 있으므로 재고 판단에서 제외)

---

### 5-5. 생산라인 (Production Line)

단일 생산라인, FIFO 스케줄링으로 재고 부족 주문을 순차 생산.

#### 핵심 계산 공식
```
부족분       = 주문 수량 - 물리적 재고(실시간 반영 포함)
실 생산량    = ceil(부족분 / (수율 * 0.9))
총 생산 시간 = 평균 생산시간(min/ea) * 실 생산량
```

수율에 0.9를 곱하는 이유: 공정 오차(불량률 10%) 추가 반영

#### 생산 완료 후 처리
- 주문 상태: PRODUCING → CONFIRMED 자동 전환
- 재고가 충분해진 시점에 출고 처리 가능

#### 내부 데이터 갱신 vs 화면 출력 (구두설명)
- 생산 진행 데이터(경과 시간, 생산 완료 여부 등)는 내부적으로 지속 계산되어 업데이트되어야 한다.
- 단, 화면(콘솔) 출력은 사용자가 해당 메뉴에 진입하는 시점에만 수행하면 된다.
- 즉, 실시간 화면 자동 갱신은 불필요하며, 메뉴 진입 시 현재 상태를 계산하여 출력하는 방식으로 구현한다.

#### 시간 처리 및 테스트 가능성 (구두설명)
- 시간은 실시간으로 처리한다.
- 단, 테스트 시에는 시간을 MOCK 등으로 설정하여 테스트 가능한 구조로 구현해야 한다.
- 구현 예시: Clock 인터페이스를 추상화하고, 실제 환경에서는 SystemClock, 테스트 환경에서는 FakeClock/MockClock을 주입하는 방식 사용.

#### 화면 표시 항목
- 현재 생산 중인 시료 정보: 주문번호, 시료명, 주문량, 재고, 부족분, 실생산량, 진행률, 완료 예정시간
- 대기 큐 목록 (FIFO 순): 순서, 주문번호, 시료명, 주문량, 부족분, 실생산량, 예상 완료 시간

---

### 5-6. 출고 처리 (Release)

#### 처리 조건
- CONFIRMED 상태의 주문만 출고 처리 가능

#### CONFIRMED 상태의 의미 (구두설명)
- CONFIRMED 상태 자체가 "재고 확보 완료"를 의미한다. 출고 시점에 재고를 별도 재확인하지 않는다.
- CONFIRMED에 도달하는 두 가지 경로:
  1. 승인 시점에 재고가 충분 → 즉시 CONFIRMED 전환
  2. 재고 부족으로 PRODUCING 전환 → 생산 완료 후 CONFIRMED 전환
- 따라서 출고 처리는 CONFIRMED 주문을 RELEASE로 전환하기만 하면 된다.

#### 출고 처리 가능 시점 (구두설명)
- 출고 처리는 재고 수량과 상관없이, 해당 주문에 대해 시작된 생산이 **완전히 완료**되어 CONFIRMED 상태가 된 이후에만 실행할 수 있다.
- PRODUCING 중인 주문은 생산이 끝나 CONFIRMED로 전환될 때까지 출고 처리 불가.
- 재고의 실시간 생산량 반영(예정 입고 수량 비례 계산)은 **신규 주문 승인 시점에만** 사용한다. 출고 처리 시점에는 이 계산을 적용하지 않는다.

#### 처리 결과
- 주문 상태: CONFIRMED → RELEASE 전환
- 표시 정보: 주문번호, 출고 수량, 처리 일시

## 6. 미션 구성

### 미션1: PoC 개발

기능 구현 전 개념 검증(Proof of Concept) 코드 작성.
각 PoC는 **별도 Repository**로 제출.

#### Repository 및 솔루션 구성 (구두설명)
- 각 PoC 항목마다 별도의 Repository를 생성해야 한다.
- 솔루션(프로젝트) 역시 각 PoC 항목별로 각각 생성하는 것을 권장한다.
- 데이터 모니터링 Tool이 데이터 영속성 처리 PoC를 참조(디펜던시)하는 것은 허용된다. 단, Repository는 반드시 분리하여 별도로 생성해야 한다.

| PoC 항목 | 설명 | Repository 이름 |
|----------|------|-----------------|
| MVC 스켈레톤 코드 | Model/Controller/View 패키지 구조와 역할 분리 완성 | ConsoleMVC-Nickname |
| 데이터 영속성 처리 | **개인별**로 선택한 방식(파일/JSON/DB 등)으로 데이터 저장·불러오기, CRUD 포함. **(구두설명) 단일 데이터(엔티티 하나) 위주로 구현** | DataPersistence-Nickname |
| 데이터 모니터링 Tool | 저장된 데이터 상태를 콘솔에서 실시간 조회하는 관리자 도구. **(구두설명) 전체 데이터 조회 위주로 구현** | DataMonitor-Nickname |
| Dummy 데이터 생성 Tool | 테스트용 Dummy Data 생성 후 연결된 DB에 추가 | DummyDataGenerator-Nickname |

* 데이터 영속성: 앱 재시작 후에도 데이터가 유지되는 성질

#### 데이터 영속성 처리 방식 (구두설명)
- 본 과제는 개인 프로젝트이므로 저장 방식(파일, JSON, DB 등)은 개인이 자유롭게 선택한다. (PDF 원문의 "팀별" 표현은 개인별로 적용)
- 요구 수준: 프로그램 종료 후에도 저장된 데이터가 남아있고, 다음 실행 시 불러올 수 있으면 충분하다. 특정 기술이나 방식을 강제하지 않는다.
- 데이터 영속성 처리 PoC는 **단일 데이터(엔티티 하나)** 위주로 구현하면 된다.
- 데이터 모니터링 Tool PoC는 **전체 데이터 조회** 위주로 구현하면 된다.
- 두 PoC는 역할이 다르므로 구현 범위를 명확히 분리할 것.

- 실 생산량: ceil(170 / (0.92 * 0.9)) = ceil(170 / 0.828) = ceil(205.3) = 206 ea
- 총 생산 시간: 0.8 min/ea * 206 ea = 164.8 min ≈ 165 min
