[PAGE 4] CHAPTER 01 - 개인과제 개요 (섹션 표지)
반도체 시료 생산주문관리 개인과제

--------------------------------------------------------------------------------
[PAGE 5] 배경
여기 가상의 반도체 회사 "S-Semi" 가 있습니다.
이 회사는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체,
대학 연구실 등의 고객에게 납품하고 있습니다.

시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고됩니다.
그런데 최근 들어 주문량이 급증하면서 문제가 생겼습니다.

문제 상황:
  "어, 이 주문 처리됐나요?"
  "공정 예약을 했는데, 언제 완성되는지 모르겠어요."
  "이미 충분한 시료 재고가 있는데, 왜 추가 공정이 돌아가고 있나요?"

엑셀과 메모장으로 주문을 관리하다 보니, 실수가 잦고 재고와 공정 현황을 한눈에 파악하기
어려웠습니다.
이러한 이유로 S-Semi에서는 더 체계적인 시료 관리를 위한
"반도체 시료 생산주문관리 시스템"을 개발하기로 결정했습니다.

--------------------------------------------------------------------------------
[PAGE 6] 역할별 흐름도 (다이어그램)

[흐름도 구성]

  고객              주문 담당자            생산 담당자
(시료 요청자)      (주문서 관리)          (시료 생산·승인)

  [고객] ---시료 요청---> [주문 담당자] ---주문서 전달---> [생산 담당자]
                               ^                               |
                               |        승인 / 거절            |
                               +-------------------------------+

흐름 설명:
  1. 고객이 주문 담당자에게 시료를 요청
  2. 주문 담당자가 요청에 맞게 주문서를 작성하여 생산 담당자에게 전달
  3. 생산 담당자가 주문을 검토 후 승인 또는 거절 결과를 주문 담당자에게 회신

역할별 행동:
- 고객        : 필요한 시료를 이메일로 요청
- 주문 담당자  : 요청에 맞게 주문서 작성
- 생산 담당자  : 개발 시료 등록 / 주문 수신 후 승인 또는 거절

--------------------------------------------------------------------------------
[PAGE 7] 시스템 개요

생산 라인:
  - 공장에서 시료 하나를 생산하는 설비 흐름
  - 하나의 생산 라인은 시료를 하나씩 생산
  - 주문이 들어온 시료에 대해서만 생산

시스템은 콘솔 기반으로 동작
  담당자가 직접 명령을 입력
  시료를 등록하고 주문을 처리

[주문 상태 흐름도 - 수영 레인(Swim Lane) 다이어그램]

  레인 구성 (위→아래):
    1) 주문 담당자 / 생산 담당자  (하나의 공유 레인)
    2) 생산 라인                  (설비 공정 레인)
    3) 고객                       (수령 레인)

  ┌──────────────────────────────────────────────────────────────────┐
  │ 주문 담당자 / 생산 담당자                                          │
  │                                                                  │
  │   주문 등록                                                       │
  │  ┌──────────┐                                                    │
  │  │ RESERVED │  ← 주황색(orange) 배지                             │
  │  └────┬─────┘                                                    │
  │       ↓                                                          │
  │   ◇ 승인 여부? ──── 거절 ────→ ┌──────────┐                     │
  │       │                        │ REJECTED │ ← 빨간색(red) 배지   │
  │      승인                      └──────────┘                     │
  │       ↓                                                          │
  │   ◇ 재고 확인?                                                    │
  │       │                재고 충분                                  │
  │   재고 부족              ↓                                        │
  │       │           ┌──────────┐   출고 처리   ┌──────────┐        │
  │       │           │CONFIRMED │──────────────→│ RELEASED │        │
  │       │           └──────────┘               └────┬─────┘        │
  │       │         ↑  (밝은 초록, bright green)       │              │
  │       │         │  점선: 생산 완료 후 복귀          │              │
  ├───────┼─────────┼──────────────────────────────────┼─────────────┤
  │ 생산 라인       │                                  │             │
  │       ↓         │                                  │             │
  │  생산 요청       │                                  │             │
  │  ┌──────────┐   │                                  │             │
  │  │PRODUCING │---┘ (생산 완료 - - - - - 점선 - - →) │             │
  │  └──────────┘                                      │             │
  │  (짙은 초록, dark green)                            │             │
  ├────────────────────────────────────────────────────┼─────────────┤
  │ 고객                                               │             │
  │                                                    ↓             │
  │                                              고객 도달            │
  │                                            (보라색 원형 노드)     │
  └──────────────────────────────────────────────────────────────────┘

흐름 요약:
  1. [주문 담당자] 주문 등록 → RESERVED (주황)
  2. [생산 담당자] 승인 여부 판단
     - 거절 → REJECTED (빨강)  ※ 모니터링 제외
     - 승인 → 재고 확인
  3. [생산 담당자] 재고 확인 분기  ※ (구두설명) 재고 확인 단계에서만 실시간 처리 적용
     - 재고 확인 시, 현재 생산 중인 생산량(PRODUCING 주문의 예정 입고 수량)을
       재고에 반영하여 가용 재고를 산출한다. (다이어그램 상 이 단계에만 해당)
     - 재고 충분 → CONFIRMED (밝은 초록) → 출고 처리 → RELEASED
     - 재고 부족 → [생산 라인] 레인으로 이동 → PRODUCING (짙은 초록)
  4. [생산 라인] 생산 완료 후 점선으로 [주문 담당자/생산 담당자] 레인의 CONFIRMED 복귀
  5. RELEASED → [고객] 레인 → 고객 도달 (보라색)

상태 배지 색상 정리:
  RESERVED  : 주황/노란색  (orange)
  REJECTED  : 빨간색       (red)
  PRODUCING : 짙은 초록색  (dark green)
  CONFIRMED : 밝은 초록색  (bright green)
  RELEASED  : 초록색       (green)
  고객 도달  : 보라색 원형  (purple)

[PAGE 8] 주문 상태 흐름

모든 주문은 아래의 상태를 보유
REJECTED는 거절된 주문으로 정상 흐름 외의 상태이며 모니터링에서 제외

상태        | 의미
------------|----------------------------------------------
RESERVED    | 주문 접수
REJECTED    | 주문 거절
PRODUCING   | 주문 승인 완료 및 재고 부족으로 생산 중
CONFIRMED   | 주문 승인 완료 및 출고 대기 중
RELEASE     | 출고 완료

[PAGE 16] 주문 승인/거절

접수된 주문(RESERVED) 목록을 확인
특정 주문에 대하여 승인 혹은 거절

접수된 주문 목록:
  - RESERVED 상태의 주문 목록 Display

※ (정책) 재고 판단 기준 : 재고 확인은 실제(물리적) 재고만을 기준으로 한다.
          PRODUCING 중인 생산량은 재고로 산입하지 않는다.
          부족분 = 주문 수량 - 실제 재고

주문 승인:
  - 접수된 특정 주문에 대해 승인
  - 승인시 재고 상황에 따라 2가지 방식으로 자동으로 처리
    a) 재고가 충분한 경우 → 주문을 즉시 CONFIRMED 상태로 전환
    b) 재고가 부족한 경우 → 생산 라인에 자동으로 등록, 주문 상태를 PRODUCING으로 전환
  ※ (구두설명) 재고 확인 시 가용 재고는 물리적 현재 재고만이 아니라,
     현재 생산 중인 수량(PRODUCING 상태 주문의 실생산량)도 재고로 포함하여 계산한다.
     가용 재고 = 현재 물리적 재고 + PRODUCING 중인 예정 입고 수량 합계
     필요 수량(이미 확보되어야 할 수량) = CONFIRMED 주문 수량 합계 + PRODUCING 주문 수량 합계
     → 가용 재고 >= 새 주문 수량 이면 CONFIRMED, 미만이면 PRODUCING
  ※ (구두설명) 예정 입고 수량은 생산 시간에 비례하여 계산한다.
     모든 생산은 일정한 속도로 이루어지며, 시간에 비례하여 제품이 생산되는 것으로 가정한다.
     예정 입고 수량(PRODUCING 주문 1건) = 실생산량 * (경과 시간 / 총 생산 시간)
     (단, 경과 시간 > 총 생산 시간인 경우 실생산량 전체로 처리)
  ※ (구두설명) 생산 여부(재고 충분/부족 분기)는 주문 승인 시점에만 판단한다.
     이미 PRODUCING 또는 CONFIRMED 상태로 전환된 이후에는 재고 변동이 생겨도
     해당 주문의 상태를 소급하여 재계산하지 않는다.
  ※ (구두설명) 생산 수량 또한 주문 승인 시점에 계산된 실 생산량으로 고정된다.
     생산 진행 중 실제 생산량이 달라지더라도 승인 시점에 결정된 수량대로 진행하며,
     생산 중 추가로 생산된 수량은 주문 시점에 반영된 내용이 아니므로 무시한다.

주문 거절:

  - 즉시 REJECTED 상태로 전환

[PAGE 20] 생산라인

생산라인에 대한 정보를 Display
주문량에 대한 부족분을 생산하되, 수율 및 오차를 고려하여 시료를 생산

[핵심 공식]
  실 생산량 : ceil(부족분 / (수율 * 0.9))
  총 생산 시간 : 평균 생산시간 * 실 생산량
  생산 완료시 주문상태 : PRODUCING -> CONFIRMED 변경

생산 현황 표기:
  - 현재 생산중인 시료에 대한 정보 표기
  - 표기할 정보 수준은 자율적으로 결정
    ex) 주문 정보, 현재까지의 생산량 등

대기 주문 확인:
  - 생산라인의 대기열인 생산 큐를 이용
  - 생산 작업을 대기하고 있는 목록을 출력
  - 스케줄링 전략 : FIFO

※ (구두설명) 생산 진행 데이터(경과 시간, 생산 완료 여부 등)는 내부적으로 지속 계산되어
   업데이트되어야 한다. 단, 화면(콘솔) 출력은 사용자가 해당 메뉴에 진입하는 시점에만
   수행하면 된다. 즉, 실시간 화면 자동 갱신은 불필요하며, 메뉴 진입 시 현재 상태를
   계산하여 출력하는 방식으로 구현한다.
※ (구두설명) 시간은 실시간으로 처리한다. 단, 테스트 시에는 시간을 MOCK 등으로
   설정하여 테스트 가능한 구조로 구현해야 한다. (예: Clock 인터페이스 추상화,
   테스트용 FakeClock/MockClock 주입 방식 등)

[PAGE 22] 출고 처리

CONFIRMED 상태의 주문에 대하여 출고를 처리
특정 주문에 대해 출고를 실행
주문 상태가 RELEASE로 전환

※ (구두설명) PDF 원문의 "재고가 충분해진 CONFIRMED 주문" 표현은 맥락상 아래와 같이
   해석해야 한다:
   CONFIRMED 상태에 도달한 주문은 이미 두 경로 중 하나를 거친 것이다.
     - 승인 시점에 재고가 충분하여 즉시 CONFIRMED로 전환된 주문
     - 재고 부족으로 PRODUCING이 되었다가 생산 완료 후 CONFIRMED로 전환된 주문
   즉, CONFIRMED 상태 자체가 "재고 확보 완료" 를 의미한다.
   출고 처리 시점에 별도로 재고를 재확인하는 것이 아니라,
   CONFIRMED 상태인 주문을 그대로 RELEASE로 전환하면 된다.
※ (구두설명) 출고 처리는 재고 수량과 상관없이, 해당 주문에 대해 시작된 생산이
   완전히 완료되어 CONFIRMED 상태가 된 이후에만 실행할 수 있다.
   즉, PRODUCING 중인 주문은 생산이 끝나 CONFIRMED로 전환될 때까지 출고 처리 불가.
※ (구두설명) 재고의 실시간 생산량 반영(예정 입고 수량 비례 계산)은 신규 주문 승인
   시점에만 사용한다. 출고 처리 시점에는 이 계산을 적용하지 않는다.

[PAGE 25] [미션1] PoC 개발

PoC(Proof of Concept) 를 작성하여 기능을 검증 후 프로젝트 구현을 시작
각 PoC는 개별 Repository로 생성하여 제출

※ (구두설명) 각 PoC 항목마다 별도의 Repository를 생성해야 한다.
   솔루션(프로젝트) 역시 각 PoC 항목별로 각각 생성하는 것을 권장한다.
※ (구두설명) 데이터 모니터링 Tool이 데이터 영속성 처리 PoC를 참조(디펜던시)하는 것은
   허용된다. 단, Repository는 반드시 분리하여 별도로 생성해야 한다.

* 데이터 영속성 : application을 다시 실행해도 데이터를 유지할 수 있는 성질

필요 PoC 항목         | 설명
----------------------|-------------------------------------------------------------
MVC 스켈레톤 코드      | Model / Controller / View 패키지 구조와 역할 분리 완성
데이터 영속성 처리     | 팀별로 선택한 방식(파일, JSON, DB 등)으로                  ← (구두설명) '팀별' 아닌 '개인별'
                      | 데이터를 저장·불러오는 구조 구현, CRUD 포함
                      | ※ (구두설명) 단일 데이터(엔티티 하나) 위주로 구현하면 된다.
데이터 모니터링 Tool   | 현재 저장된 데이터 상태를 콘솔에서 실시간 조회할 수 있는 관리자 도구
                      | ※ (구두설명) 전체 데이터 조회 위주로 구현하면 된다.
Dummy 데이터 생성 Tool | Test 를 위한 Dummy Data를 생성하는 도구
                      | Dummy Data는 연결된 DB에 추가

※ (구두설명) 본 과제는 개인 프로젝트이므로, 데이터 영속성 처리 방식(파일, JSON, DB 등)은
   팀이 아닌 개인이 자유롭게 선택한다.
※ (구두설명) 데이터 영속성의 요구 수준은, 프로그램이 종료된 후에도 저장된 데이터가
   남아있고 다음 실행 시 불러올 수 있으면 충분하다. 특정 저장 기술이나 방식을 강제하지 않는다.
